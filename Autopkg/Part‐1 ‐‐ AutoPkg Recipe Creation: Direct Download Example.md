# AutoPkg + Jamf Upload Steps

## Overview

Uploading a package to Jamf involves **three sequential AutoPkg recipes**:

1. **Download Recipe** – Downloads the application installer (.dmg, .pkg, .zip)
2. **PKG Recipe** – Creates a .pkg from the downloaded app if needed
3. **Jamf Recipe** – Uploads the package to Jamf and creates Smart Groups / Policies

The recipes are organized in a **parent-child** hierarchy:

```text
download.recipe
  └── pkg.recipe
        └── jamf.recipe
```

Before writing any recipes, it’s important to:
- Understand the application download process
- Capture a valid download URL
- Determine if authentication or scraping is needed

---

## 1. Understanding the Download Flow

Some vendor URLs are **direct and static**, while others require authentication or JavaScript-based navigation.

### Example: Static Direct Download

```text
https://dl.google.com/chrome/mac/universal/stable/GGRO/googlechrome.dmg
```

- This URL is simple and versionless.
- No scraping or authentication is needed.
- It rarely changes.

### Using `URLDownloader`

You can use the `URLDownloader` processor to fetch the installer.

```yaml
- Processor: URLDownloader
  Arguments:
    filename: 'googlechrome.dmg'
    url: '%DOWNLOAD_URL%'
```

**Important Input and Output Variables:**
- `filename`: You control what name the file is saved as in the downloads folder.
- `pathname`: Full path to the downloaded file; generated by AutoPkg.
- `last_modified`: HTTP Last-Modified header value from the server.
- `etag`: HTTP ETag header value.
- `download_changed`: Boolean indicating if this run downloaded a new version.

All of these variables are set in the parent recipe and passed to child recipes automatically.

---

## 2. Extracting Version from the App

Use the `Versioner` processor to extract the version from the mounted .dmg:

```yaml
- Processor: Versioner
  Arguments:
    input_plist_path: '%pathname%/Google Chrome.app/Contents/Info.plist'
```

- This mounts the `.dmg` (if applicable) and reads version info from `Info.plist`.
- It sets the `version` environment variable, which is available for downstream use in `.pkg` and `.jamf` recipes.

> Note: If your file is a `.pkg`, you might use `PkgPayloadVersioner` or similar instead.

---

## 3. Code Signature Verification

To verify the integrity of the downloaded application:

### Get the Designated Requirement:

```bash
codesign -dr- "/Volumes/Google Chrome/Google Chrome.app"
```

Use the output with `CodeSignatureVerifier`:

```yaml
- Processor: CodeSignatureVerifier
  Arguments:
    input_path: '%pathname%/Google Chrome.app'
    requirement: 'identifier "com.google.Chrome" and anchor apple generic and certificate ...'
```

- Ensures the downloaded app is signed correctly and hasn’t been tampered with.
- If verification fails, AutoPkg will stop processing.

---

## 4. Optional: Stop Further Processing

Use `EndOfCheckPhase` to stop processing after download/version check:

```yaml
- Processor: EndOfCheckPhase
```

- Common in shared download recipes
- Prevents unnecessary package creation or upload if only checking for updates

---

## Example: Google Chrome Download Recipe

```yaml
Description: Downloads the latest version of Google Chrome
Identifier: com.github.adtejaa-actions.download.GoogleChrome
MinimumVersion: '2.3'

Input:
  NAME: Google Chrome
  SOFTWARE_TITLE: '%NAME%'
  DOWNLOAD_URL: https://dl.google.com/chrome/mac/universal/stable/GGRO/googlechrome.dmg

Process:
  - Processor: URLDownloader
    Arguments:
      filename: 'googlechrome.dmg'
      url : '%DOWNLOAD_URL%'

  - Processor: Versioner
    Arguments:
      input_plist_path: '%pathname%/Google Chrome.app/Contents/Info.plist'

  - Processor: EndOfCheckPhase

  - Processor: CodeSignatureVerifier
    Arguments:
      input_path: '%pathname%/Google Chrome.app/'
      requirement: 'identifier "com.google.Chrome" and anchor apple generic and certificate ...'
```

# How to run the download recipe????

This guide outlines the steps to create and work with an AutoPkg download recipe and its override. This applies when you write a recipe from scratch and want to test or run it securely.

---

## ✅ Step 1: Create a New Download Recipe
After creating the recipe, save it. (e.g., `googlechrome.download.recipe.yaml`). This file can reside anywhere, such as your `Documents` folder. Since 

---

## ✅ Step 2: Create an Override
Overrides are stored in `~/Library/AutoPkg/RecipeOverrides` and are required for trust and verification.

```bash
autopkg make-override /path/to/googlechrome.download.recipe.yaml
```

📌 **Note:** You can also create an override using the identifier:
```bash
autopkg make-override com.github.adtejaa-actions.download.GoogleChrome
```

```bash
output:
Override file saved to /Users/<yourname>/Library/AutoPkg/RecipeOverrides/GoogleChrome.download.recipe
```

## ✅ Step 3: Trust the Override
When the override is created, it has no trust info by default. You need to update trust:

```bash
autopkg update-trust-info googlechrome.download.recipe
```

✅ This will embed trust metadata into the override file.

---

## ✅ Step 4: Run the Override Recipe

Now you can safely run it without trust errors:

```bash
autopkg run googlechrome.download.recipe
```

📌 Alternatively, you can also run:

```bash
autopkg run -vvvv local.download.googlechrome
```

Or

```bash
autopkg run GoogleChrome.download.recipe
```

📁 The override file is stored at:

```bash
~/Library/AutoPkg/RecipeOverrides/googlechrome.download.recipe
```
